# PREMORTEM
## "이 프로젝트는 왜 실패했는가" — 사전 부검 보고서

> **Premortem이란?** 프로젝트가 실패했다고 가정하고, 역방향으로 실패 원인을 찾는 전략적 사고 도구.  
> Gary Klein이 제안한 이 방법론은 낙관적 편향을 걷어내고 숨겨진 위험을 드러낸다.

작성일: 2025년 6월  
대상 프로젝트: 무신사 러닝화 리뷰 기반 AI 추천 시스템 (11조)

---

## 가상 시나리오 설정

> "2026년 3월, 우리 서비스는 출시 6개월 만에 사실상 사용 중단되었다. 무신사 측은 기능을 비활성화했고, 팀은 해산했다. 무엇이 잘못되었는가?"

---

## 실패 시나리오 1: 데이터 품질이 기대에 미치지 못했다

### 발생 원인
- 무신사 러닝화 카테고리의 리뷰 수가 충분하지 않은 제품이 다수 존재
  - 신제품, 비주류 브랜드(브룩스, 써코니 등)는 리뷰 수십 건 수준
  - 통계적으로 유의미한 속성 점수를 산출하려면 제품당 최소 수백 건 이상 필요
- 리뷰 내용이 너무 짧거나 의미 없는 경우 다수
  - "빠른 배송 감사합니다", "선물용입니다" 등 제품 속성과 무관한 리뷰
  - 이런 리뷰가 전체의 30~40%에 달할 수 있음
- 크롤링/API 수집 과정에서 데이터 편향 발생
  - 특정 시즌, 특정 이벤트(할인) 기간에 쏠린 리뷰

### 영향
- 일부 제품의 속성 프로필이 통계적으로 무의미하거나 왜곡됨
- "쿠션감: ★★★★★"인데 실제 사용자들은 쿠션에 불만인 역설 발생
- 사용자 신뢰 붕괴

### 대응책 (예방)
- [ ] 제품당 최소 리뷰 수 임계값 설정 (예: 50건 미만은 "데이터 부족"으로 표시)
- [ ] 리뷰 필터링 로직 강화: 길이 30자 미만, 배송/포장 관련 키워드 포함 리뷰 제외
- [ ] 리뷰 수집 주기를 월 1회 이상으로 유지하여 신규 리뷰 반영
- [ ] 신제품의 경우 브랜드 공식 스펙 데이터와 결합하여 보완

---

## 실패 시나리오 2: 감성 분석 모델의 정확도가 실사용에서 낮았다

### 발생 원인
- **반어법/비꼬는 표현 미처리**
  - "이 신발 정말 '가벼워요'... 신을 수가 없어서요" → 긍정으로 잘못 분류
  - KNU SentiLex 기반 규칙 모델은 맥락 이해 불가
- **브랜드 특수 용어 미처리**
  - "부스트 쿠션", "에너지 리턴", "카본 플레이트" 등 러닝화 전문 용어가 감성 사전에 없음
  - 이 단어들이 포함된 리뷰의 속성 점수가 0으로 계산
- **신조어/줄임말 처리 실패**
  - "갓성비", "레전드", "존예", "개 좋음" 등 MZ 세대 언어가 감성 사전에 없음
- **속성 키워드 오분류**
  - "가격 대비 쿠션이 아쉽다" → "가격" 속성인가, "쿠션" 속성인가 판단 모호
  - 하나의 문장이 두 속성에 걸쳐 있을 때 처리 기준 불명확

### 영향
- 제품 A의 쿠션감이 실제로 나쁜데 "쿠션감: +2.1"로 표시
- 사용자가 추천받은 제품을 사고 실망 → 신뢰 하락 → 재이용 거부

### 대응책 (예방)
- [ ] 분석 결과에 대한 정확도 검증: 전문 러너 5~10인이 수동으로 제품 프로필 검토
- [ ] 부정 전치사 처리 규칙 추가 ("별로", "아쉽다", "기대 이하" 앞의 감성 역전)
- [ ] 러닝화 도메인 특화 감성 사전 구축 (브랜드 용어, 스펙 용어 추가)
- [ ] 향후 BERT 기반 모델로 단계적 전환 계획 수립
- [ ] A/B 테스트: 규칙 기반 vs BERT 기반 추천 품질 비교

## 실패 시나리오 3: 사용자가 AI 추천을 신뢰하지 않거나 사용하지 않았다

### 발생 원인
- **추천 이유가 불투명하다고 느낌**
  - "아디다스 울트라부스트 추천" — 왜 이 제품인지 납득 어려움
  - "87% 일치"라는 숫자가 어디서 나온 건지 모름
- **추천 결과가 이미 알고 있는 인기 제품과 동일**
  - 데이터가 많은 인기 제품이 항상 추천에 등장 → "이건 그냥 베스트셀러잖아요"
- **AI 추천 모달의 자유형 입력이 낯설거나 번거로움**
  - 소비자는 텍스트 입력보다 필터 클릭에 익숙
  - "뭐라고 써야 할지 모르겠다"는 반응
- **개인정보/데이터 수집에 대한 거부감**
  - "내 취향 데이터를 무신사가 수집하나요?"라는 우려

### 영향
- AI 추천 기능 클릭률(CTR) 3% 미만 → 비활성화 결정
- "있으나 마나한 기능"으로 전락

### 대응책 (예방)
- [ ] 추천 이유를 한국어로 명확하게 표시: "이 제품은 리뷰 347건 기준 쿠션감이 높고 발볼이 넓다는 언급이 많습니다"
- [ ] 속성 필터(무게감/쿠션감/발볼넓이)를 메인 추천 진입점으로 강화
- [ ] AI 모달에 예시 프롬프트 제공: "예) 평발이고 하프 마라톤을 준비 중이에요"
- [ ] 개인화 온보딩 3단계 설계 (발 유형 → 러닝 목적 → 선호 속성)
- [ ] 초기 사용자 100명 대상 UX 테스트로 진입 장벽 파악

## 실패 시나리오 4: 무신사 플랫폼 통합에 기술적 장벽이 있었다

### 발생 원인
- **무신사 API 접근 불가**
  - 공식 파트너십 없이는 제품 데이터, 리뷰 데이터 실시간 수집 불가
  - 크롤링은 이용약관 위반 가능성 + 지속 가능성 낮음
- **프론트엔드 통합 거절**
  - 무신사 측이 외부 팀의 JS 코드를 자사 서비스에 삽입 거부
  - 무신사 앱 내 WebView 형태 통합도 불확실
- **실시간 추천 API 레이턴시 문제**
  - 속성 프로필 계산을 실시간으로 하면 응답 시간 5초 이상
  - 사용자는 3초 이내 응답을 기대
- **리뷰 데이터 저작권 문제**
  - 무신사 리뷰는 무신사 소유 → 분석에 사용하는 것에 법적 이슈 가능성

### 영향
- 아무리 좋은 모델을 만들어도 배포할 채널이 없음
- 독립 서비스로 전환 시 트래픽 확보 어려움

### 대응책 (예방)
- [ ] 무신사와 공식 데이터 파트너십 제안서 준비 (MOU 또는 API 라이선스)
- [ ] 초기에는 공개 데이터만 활용한 Proof of Concept 구축
- [ ] 백엔드에서 속성 프로필을 사전 계산(pre-compute) → DB에 캐시 → API 응답 100ms 이내
- [ ] 법무팀 검토: 공개 리뷰의 분석 이용 적법성 확인
- [ ] 플랜 B: 무신사 외 독립 서비스 또는 크롬 익스텐션 형태 검토

## 실패 시나리오 5: 경쟁사가 더 빠르게 유사 기능을 출시했다

### 발생 원인
- 무신사 자체 AI 추천팀이 유사 기능을 3개월 내 출시
- 네이버 쇼핑이 하이퍼클로바 기반 리뷰 분석 추천 기능 적용
- 스타트업이 더 정교한 러닝화 전문 추천 앱 출시

### 영향
- 차별화 포인트 상실 → 우리 서비스의 존재 이유 약화
- 무신사가 자체 개발로 전환하여 파트너십 해지

### 대응책 (예방)
- [ ] 특허 또는 선점 발표로 기술적 우선성 확보
- [ ] 무신사 독점 계약 협상 (1년 배타적 파트너십)
- [ ] 러닝화 → 등산화 → 골프화 → 의류 등 빠른 카테고리 확장으로 방어막 형성
- [ ] 속성 사전, 학습 데이터 등 독자 자산 구축으로 진입 장벽 조성

---

## 실패 시나리오 6: 팀 내부 역량 부족 또는 분열

### 발생 원인
- NLP/ML 전담 인력 부재 → 모델 품질 정체
- 프론트엔드/백엔드 통합 경험 없음 → 개발 지연
- 데이터 수집, 분석, 서비스 개발을 5명이 동시에 담당 → 번아웃
- 역할 분담 불명확 → 책임 공백

### 영향
- 핵심 기능 미완성 상태로 출시 → 초기 인상 훼손 → 회복 불가

### 대응책 (예방)
- [ ] RACI 매트릭스로 역할 및 책임 명문화
- [ ] 각 팀원의 핵심 기술 영역 정의 및 집중
- [ ] 주 1회 스프린트 리뷰 + 2주 단위 데모
- [ ] 외부 기술 자문(KAIST 지도교수, NLP 전문가) 정기 멘토링

---

## 실패 시나리오 7: 모델이 특정 브랜드/제품에 편향되었다

### 발생 원인
- 무신사 내 나이키, 아디다스 등 대형 브랜드가 리뷰 수 압도적으로 많음
- 결과적으로 이 브랜드 제품들의 속성 프로필이 더 정확하고 신뢰도 높음
- 소규모 브랜드(브룩스, 써코니, 미즈노)가 추천에서 체계적으로 불이익

### 영향
- 추천이 결국 "인기 브랜드 다시 추천"으로 수렴 → 기존 베스트셀러 목록과 차별화 없음
- 소규모 브랜드 파트너사 반발 가능성

### 대응책 (예방)
- [ ] 리뷰 수 기반 신뢰도 가중치 도입 (리뷰 많은 제품 = 더 신뢰할 수 있는 프로필)
- [ ] 리뷰 부족 제품에 대해서는 추천 보류 또는 "신뢰도 낮음" 표시
- [ ] 장기적으로 크라우드소싱을 통한 전문 러너 리뷰 수집 프로그램 운영

## 핵심 교훈 (Premortem Summary)

1. **데이터 없이는 모델도 없다**: 리뷰 수 임계값과 품질 필터가 첫 번째 방어선
2. **추천의 이유를 설명하라**: 블랙박스 추천은 신뢰를 쌓지 못한다
3. **플랫폼 파트너십이 기술보다 먼저다**: 아무리 좋은 모델도 배포 채널 없이는 무의미
4. **정확도는 사용성보다 우선이다**: 80% 정확한 서비스가 60% 정확한 서비스보다 낫고, 출시 안 하는 것보다 낫다
5. **MVP를 최대한 빨리 검증하라**: 2~3개 제품만으로도 파이프라인 전체를 검증할 수 있다

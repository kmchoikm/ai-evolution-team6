# Premortem: 실패 시나리오 및 사전 방어 전략

**가정: 출시 후 1년, Save & Grow는 활성 사용자 확보에 실패하고 서비스를 종료하게 되었다. 그 원인은 무엇일까?**

## 🚨 시나리오 1: 대형 핀테크(토스/뱅크샐러드)의 '가족 기능' 출시 (Risk: High)
* **실패 원인:** 런칭 6개월 후, 토스가 '가족 고정비 연동' 기능을 업데이트했다. 사용자는 굳이 새로운 앱을 깔 필요를 느끼지 못했다.
* **예방 조치 (대응책):** * 단순 데이터 조회로는 승부할 수 없다. **'AI 협상 스크립트'와 '자동 해지 딥링크'**라는 행동 중심의 강력한 킬러 피처를 초기에 뾰족하게 각인시켜야 한다.
    * 커뮤니티적 요소(실제 혜택 협상 성공 사례 공유)를 강화하여 플랫폼 이탈을 방어한다.

## 🚨 시나리오 2: 온보딩 피로도로 인한 초기 이탈률 80% (Risk: High)
* **실패 원인:** 마이데이터 연동 인증 과정이 너무 길었고, 특히 "배우자의 데이터를 불러오기 위한 동의" 과정에서 UX가 단절되어 대다수가 앱을 삭제했다.
* **예방 조치 (대응책):** * 온보딩 단계에서 마이데이터 연동을 뒤로 미룬다. 
    * 나이/성별/자녀 유무만 묻는 **'초간편 심리/행동 테스트'**로 시작하여, "당신이 놓치고 있는 예상 금액은 월 12만 원"이라는 호기심을 유발한 뒤 가입을 유도한다.

## 🚨 시나리오 3: 투자 연결 기능의 법적 제재 (Risk: High)
* **실패 원인:** '절약한 돈을 특정 ETF로 추천해 주는 기능'이 금융당국으로부터 미인가 투자자문업으로 해석되어 서비스 중단 명령을 받았다.
* **예방 조치 (대응책):** * 기획 초기 단계부터 핀테크 전문 법무법인 검토 필수 진행.
    * '종목 추천'이 아닌 '포트폴리오 시뮬레이션 정보 제공'으로 UI/UX 워딩을 엄격히 제한하고, 실제 매매는 제휴 증권사 앱으로 아웃링크 처리한다.

## 🚨 시나리오 4: 협상 대본의 무용지물화 (Risk: Medium)
* **실패 원인:** AI가 짜준 스크립트대로 통신사에 전화했으나, 기업들이 정책을 바꿔 더 이상 해지 방어 혜택(상품권 등)을 주지 않아 유저가 실망했다.
* **예방 조치 (대응책):** * 사용자들로부터 '협상 성공/실패' 데이터를 실시간 피드백 받아 AI 모델을 지속 업데이트한다.
    * "현재 이 스크립트의 최근 1주일 성공률은 45%입니다"와 같이 투명하게 확률을 고지하여 기대감을 관리한다.

\##  Premortem: Why This Project Might Fail



이 프로젝트가 1년 후 실패했다고 가정하고,

그 이유를 미리 분석한다.



\---



\##  1. 사용자가 실제로 행동하지 않는다



\### 문제

\- 추천은 보지만 실행하지 않음

\- 협상, 해지 귀찮음



\### 대응

\- 자동화 기능 강화

\- “원클릭 실행” 구조

\- 알림 + 리마인드 시스템



\---



\##  2. 데이터 확보 문제



\### 문제

\- 금융/지출 데이터 연동 어려움

\- API 제한



\### 대응

\- 초기에는 수동 입력 기반

\- 점진적 API 확장

\- MVP는 단순 구조로 시작



\---



\##  3. 차별성이 부족하다



\### 문제

\- 기존 가계부 앱과 유사



\### 대응

\- “절약 → 투자 자동 연결”에 집중

\- 기능 축소하고 핵심 강화



\---



\##  4. AI 필요성이 약하다



\### 문제

\- 단순 rule-based로도 가능



\### 대응

\- 개인 맞춤 추천 강화

\- 행동 유도 메시지 생성

\- 투자 전략 개인화



\---



\##  5. 투자 기능 규제 리스크



\### 문제

\- 투자 추천은 법적 제한 존재



\### 대응

\- 직접 투자 권유 대신

\- “교육 + 시뮬레이션 중심”

\- ETF 기반 일반 전략 사용



\---



\##  6. 사용자 유지율 낮음



\### 문제

\- 초기에만 사용하고 이탈



\### 대응

\- 지속적 가치 제공

\- 월간 리포트

\- 자산 성장 시각화



\---



\##  7. 너무 많은 기능



\### 문제

\- 복잡함 → 사용성 저하



\### 대응

\- MVP는 1\~2 기능 집중

\- 핵심: 절감 + 투자 연결



\---



\##  핵심 교훈



이 서비스의 실패는 대부분



&#x20;“사용자가 행동하지 않는 것”에서 시작된다
